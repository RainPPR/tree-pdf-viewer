## 3) 我给你的“极致性能”落地架构（PDFium 版）

SwapChainPanel 的定位非常明确：承载 DirectX 渲染结果，并在 XAML UI 中呈现。 [11](https://learn.microsoft.com/en-us/windows/windows-app-sdk/api/winrt/microsoft.ui.xaml.controls.swapchainpanel?view=windows-app-sdk-1.8)

### 3.2 PDF 渲染管线（让滚动/缩放丝滑的关键）

做“像 VSCode 一样顺滑”的 PDF 阅读器，本质不是“单次渲染快”，而是 **“永远只做对用户当前视口最有价值的渲染”**：

- 
**布局阶段（Layout）**

- 先快速拿到页尺寸、页数、目录（outline）等元信息（不渲染）。
- 计算每页在当前缩放下的布局位置（纵向连续流 or 单页翻页）。

- 
**视口阶段（Viewport）**

- 监听滚动位置/缩放倍率 → 计算“当前可见页集合 + 上下预取页集合”（比如可见页 ±2 页）。
- 对每个目标页，计算目标分辨率（与缩放相关）。

- 
**切片阶段（Tiling）**（核心）

- **不要**“整页渲染成一张大位图再贴上去”。
- 采用 **固定大小 tile**（例如 512×512 或 768×768），只渲染视口覆盖到的 tile。
- 好处：

- 内存可控（tile 缓存可 LRU）
- 缩放时可以先用旧 tile 放大凑合，再逐步补高分辨率 tile（用户感知“瞬间响应”）

- 
**渲染队列（Render Queue）**

- 渲染任务按优先级：**视口内 > 视口边缘 > 预取**
- 支持取消：缩放/快速滚动时，旧任务直接 cancel（否则 CPU 会被无意义渲染吃满，造成“越滚越卡”）

- 
**展示阶段（Present）**

- tile 渲染到 CPU 内存位图后，上传到 GPU texture（D3D11）
- 用 swap chain 以 60/120Hz 呈现（取决于显示器）

### 3.3 1GB 内存上限怎么“算得过来”（非常关键的工程现实）

PDF 阅读器最容易爆内存的是**位图缓存**。你可以用一个简单的量级估算：

- BGRA 32-bit 位图：**每像素 4 字节**
- 512×512 的 tile：512×512×4 ≈ **1MB**
- 如果你缓存 300 个 tile：≈ **300MB**
- 再加上：PDF 解码内部缓存、字体、图片解码缓存、UI、纹理副本、文件索引……很容易就逼近 1GB。

所以建议你做**硬预算**（例子，给你一个可直接照抄的策略）：

- **Tile GPU 纹理缓存：300–500MB 上限（LRU）**
- **Tile CPU 位图缓存：50–150MB 上限（可选；或直接渲染到可映射纹理减少一份拷贝）**
- **预渲染页数：默认 2 页（快速滚动时降为 0）**
- **缩放中的“临时高分辨率 tile”：单独小预算（比如 100MB），缩放停止后再慢慢替换**

目标不是“渲染最快”，而是“永远不失控”：CPU 不被过期任务占满、内存不被缓存撑爆、GPU 不被无上限纹理塞满。

## 4) 并发/多线程：PDFium 的坑要提前绕开（否则你会随机崩溃）

PDFium 在很多封装/文档里都会明确提醒：**不保证线程安全（assumed not thread-safe / inherently not thread-safe）**。 [12](https://github.com/ajrcarey/pdfium-render)
因此在“极致性能”的实现里，正确姿势通常是二选一：

### 方案 A（最稳，推荐）

- **单独一个“PDF 渲染线程”**（或“每个打开文档 1 个渲染线程”）
- UI 线程只负责投递任务、收结果、贴图
- 其它 CPU 工作（文件树扫描、索引、搜索）可以并行，但**所有 PDFium 调用统一串行化**

这在交互式阅读器里非常常见：你要的是低延迟和可取消，而不是“同时渲染 20 页跑满所有核”。

### 方案 B（追求更高吞吐）

- 用 **多进程** 做并行渲染（每个进程 1 个 PDFium 实例），主进程负责 UI 与合成
这也符合一些封装对 PDFium 的建议方向（既然线程不安全，就用进程隔离）。 [4](https://pypdfium2-team.github.io/pypdfium2/python_api.html)
代价是 IPC、共享纹理/共享内存、工程复杂度上升。

反过来，如果你选 **MuPDF**，它的文档会更系统地讲多线程使用方式（比如“文档服务器线程创建 display list，其他线程渲染”等思路）。 [6](https://mupdf.readthedocs.io/en/latest/reference/c/overview.html)

## 5) 如果你想“尽量少造轮子”，这两条路线也很实际

### 路线 1：WebView2 内嵌（最快能做出一个体验不错的阅读器）

- WebView2 本质是把 Edge（Chromium）嵌入桌面应用；它有明确的多进程模型。 [1](https://developer.microsoft.com/microsoft-edge/webview2/)
- PDF 打开就直接用 Edge PDF viewer；你可以把左侧目录树/文件 tab 自己做，右侧用 WebView2 当“文档画布”。
- 但要注意：Edge 的新 PDF 体验/引擎切换（含 Adobe 引擎）会受到策略控制（例如 WebView2 的 NewPDFReaderWebView2List）。 [7](https://learn.microsoft.com/zh-cn/deployedge/microsoft-edge-webview-policies)
这意味着：你对“用户机器上到底跑什么 PDF 引擎”控制力有限。

**适合：**你更在意“马上做出来能用”，且 1GB 内存不是硬指标（只是希望别太夸张）。

### 路线 2：WinRT 的 Windows.Data.Pdf（依赖少但能力边界明显）

- 它主要是把 PDF 页“渲染成图像/流”给你用。 [10](https://learn.microsoft.com/en-us/archive/msdn-magazine/2013/december/windows-8-1-rendering-pdf-content-in-windows-store-apps)
如果你要做高级体验（分层、tile、智能缓存、文本层 overlay），你很快会发现它不如专用引擎顺手。

### 1. PDF 渲染内核推荐

- **首选：PDFium**（Google/Chromium 使用）——通过 pdfium-render（Rust 高阶绑定）或直接集成。性能优秀、支持复杂 PDF 好、更新活跃、低内存足迹表现稳定。适合大文件和多文档场景。[2](https://www.nutrient.io/blog/why-pdfium-is-a-trusted-platform-for-pdf-rendering/)
- **强力替代：MuPDF**（SumatraPDF 使用的引擎）——在很多基准测试中渲染速度最快，内存占用特别低，非常适合“轻量极致”阅读器。Artifex 官方提供了 **MuPDF.NET**（C# 绑定），在 Windows 上表现优秀。[3](https://connect.hyland.com/t5/alfresco-blog/pdf-rendering-engine-performance-and-fidelity-comparison/ba-p/125428)

**避免或谨慎使用**：纯 WebView2（即 DocBox 采用的方式）。它很流畅，但 Chromium 基线内存较高（单实例还好，多标签容易吃掉几百 MB），在严格 1GB 限制下会比较吃力，除非你只保持少量活跃标签。
**渲染实现关键（无论选哪个引擎）**：

- **按需渲染**：只渲染当前可见页面 + 少量前后预取。
- **严格 LRU 缓存**：控制渲染位图总内存（例如上限 300–500MB），大页面使用分块（tiled）渲染。
- **内存映射文件** + 后台线程渐进式渲染 + 低分辨率缩略图。
- 非活跃文档及时卸载或只保留元数据。
这些架构技巧比语言本身更能决定是否能稳定在 1GB 以内。SumatraPDF 的源码是很好的学习对象（即使你不完全照抄它的 UI）。

### 3. 整体架构建议（决定成败的关键）

- **左侧**：虚拟化 TreeView（只加载可见节点），支持文件夹监控（FileSystemWatcher 或 Rust 等价物）。
- **右侧**：TabView，每个标签对应一个文档。非活跃标签只保留最小元数据。
- **渲染控件**：自定义控件，只在可见时请求高精度渲染。
- **内存策略**：

- 全局渲染缓存上限（例如 400MB）。
- 大 PDF 使用低 DPI 预览 + 按需高 DPI 块。
- 定期清理未使用页面。
- 使用内存映射而不是全加载到 RAM。

- **其他优化**：多线程渲染（但 PDFium/MuPDF 本身线程安全需注意，通常建议进程级并行而非重度多线程）；进度显示；仅 Windows 11 时可适当用 Mica/Acrylic 效果（开销很小）。

这个架构做好后，1GB 限制下同时打开 10–20 个中等 PDF 也是可行的（极端扫描图像大文件除外）。
