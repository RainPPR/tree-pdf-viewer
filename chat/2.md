## 你真正要优化的“性能瓶颈”是什么（决定语言/渲染方案）

做 PDF 阅读器，最容易卡的不是目录树 UI，而是：

- **PDF 页面的解释 + 光栅化（render to bitmap）**：CPU 密集、还可能触发大量字体/图片解码缓存。
- **滚动/缩放时的重绘策略**：如果你每一帧都重新渲染整页，必卡；正确做法是“GPU 合成位图 + 后台补高分辨率”。
- **内存预算（1GB）**：真正吃内存的是“渲染后的位图缓存（RGBA）”和“图片解码后的中间缓存”，而不是 PDF 文件本身。

这也是为什么“语言快不快”通常不是第一决定因素——**核心是你能否把渲染与显示解耦、做对缓存/分块/渐进**。

## “极致性能”的渲染方案：正确的显示管线长什么样

下面这套思路，不管你用 MuPDF 还是 PDFium，基本都通用：

### 1) UI 只做两件事：布局 + 合成（不要做渲染）

- 右侧阅读区不要用“每页一个大控件然后让框架自己重绘”。
- 用一个自定义渲染层（SwapChain/Composition surface），UI 线程只负责：

- 当前滚动位置、缩放比
- 需要哪些页/哪些 tile
- 把“已有的位图纹理”摆到正确位置（GPU 合成）

Windows 的 Composition/Visual Layer 本来就是为“高性能、保留模式合成与动画”设计的体系。[10](https://github.com/microsoft/Windows.UI.Composition-Win32-Samples)

### 2) 渲染用“分块 + 多级分辨率 + 渐进补帧”

目标是：**滚轮一动马上 60/120fps 跟手**，哪怕页面还糊；停下来 50~150ms 内补清晰。
常用做法：

- **Low-res 预览层**：比如按屏幕 0.5x 或 1.0x 分辨率先出图
- **Hi-res 精细层**：用户停止滚动后再渲染 1.5x~2.0x（按 DPI/缩放决定）
- **Tile（分块）**：对超大页/大缩放，只渲染可视区域附近的 tile（例如 256/512/1024 像素块），其余不碰

如果你选 **Windows.Data.Pdf**，它的设计文档里就提到渲染到 SurfaceImageSource/VirtualSurfaceImageSource，本质上就是走“面向表面/虚拟表面”的思路（很接近 tile 化渲染的方向）。[11](https://learn.microsoft.com/en-us/archive/msdn-magazine/2013/december/windows-8-1-rendering-pdf-content-in-windows-store-apps)
如果你选 **PDFium**，可以用它的 progressive rendering：FPDF_RenderPageBitmap_Start/Continue 先让 UI 线程尽快拿到可用输出（或在后台把首帧做出来）。[3](https://pdfium.googlesource.com/pdfium/%2B/master/public/fpdf_progressive.h)
如果你选 **MuPDF**，强烈建议用“display list 录制 + 重放”：页面解析一次，后面不同缩放/不同裁剪区域重放列表生成 pixmap，减少重复解释开销；并且 display list 可被多线程用于渲染。[12](https://mupdf.readthedocs.io/en/1.23.0/using-mupdf.html)

### 3) GPU 负责“缩放/平移/裁剪/叠加”，CPU 负责“把 PDF 变成像素”

Direct2D/Direct3D 的常见模式是：GPU 资源（位图/纹理）用于显示合成，硬件加速可用时这些资源会映射到 GPU。[13](https://learn.microsoft.com/el-gr/windows/win32/direct2d/resources-and-resource-domains)
所以你要尽量避免“每次滚动都 CPU 重新渲染整页”，而是：

- 先把旧纹理平移（GPU 超快）
- 后台线程渲染缺失 tile → 上传纹理 → UI 下一帧合成进去

## 1GB 内存预算：怎么“可证明地”守住

### A) 渲染库内部缓存要限额（MuPDF 特别好做）

MuPDF 创建 context 时第三个参数就是 store 大小（示例里甚至出现 FZ_STORE_UNLIMITED 这种配置），也就是说它的 store 是你可控的缓存池。[14](https://mupdf.readthedocs.io/en/1.26.1/C-API-core.html)
你可以把它设置成比如 256MB~512MB，让剩余预算留给：

- 右侧纹理/位图缓存（RAM + VRAM）
- 缩略图
- 目录树数据结构、文本索引（如果你要做全文检索）

### B) 位图缓存必须做 LRU + 分级（否则必爆）

粗算一下：
一张 RGBA 位图内存 ≈ 宽 * 高 * 4 bytes。
比如你渲染到 3000×4000（较高 DPI 的整页）≈ 48MB。缓存 10 页就是 480MB（还没算 tile、多分辨率、临时缓冲）。
所以要做：

- “只缓存可视区附近 N 页/若干 tile”
- 同一页只保留 1~2 个分辨率（并可随缩放抖动动态调整）
- 大图/扫描件页要优先 tile 化，否则渲染临时峰值很容易把你顶到 1GB

### C) 最硬核的做法：渲染进程隔离 + Job Object 强制内存上限

如果你真的要“**无论什么 PDF 都不能超过 1GB**”，最稳的工程手段是：

- 主进程：UI + 目录树 + 合成
- 子进程：PDF 打开/解析/渲染
- 用 **Windows Job Object** 给子进程设置内存限制（进程级/Job 级），超限就让渲染失败并重启渲染进程（UI 不死）

Microsoft Learn 的 Job Objects 文档与相关结构体说明了 Job 可用于限制资源，并能通过 limit flags/working set 等方式约束内存行为。[15](https://learn.microsoft.com/en-us/windows/win32/procthread/job-objects)
这种架构还有一个隐藏收益：PDF 解析/渲染属于高风险攻击面（历史上 PDF 引擎漏洞很多），进程隔离天然更安全。

### 关键内存控制策略

#### 1. 虚拟化页面渲染（最重要！）

只渲染**视口内可见的页面**，超出视口的页面仅保留尺寸信息，不保留像素数据。一个 300dpi 的 A4 页面约 3MB RGBA，100页若全部解码将消耗 300MB——虚拟化后只需保留 5~10 页。

#### 2. LRU 页面纹理缓存

```
text已解码页面 → LRU Cache（设定最大 300~400MB）
              ↳ 超出限制时自动驱逐最久未访问的页面纹理
```

#### 3. 异步渲染，不阻塞 UI

使用 Rust 的 rayon 线程池在后台解码 PDF 页面，主线程只负责 UI 绘制，解码完成后通过 channel 将纹理推送到 GPU。

#### 4. 延迟加载目录树

目录树只在展开节点时才扫描子目录，不一次性加载整个文件系统树。

```
text第一阶段（骨架）：
```
