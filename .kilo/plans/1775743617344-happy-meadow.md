# Tree PDF Viewer 修复计划

## 概述

修复四个 PDF 渲染引擎的编译和运行时问题：
1. **MuPDF** - 4个编译错误 (API 0.6.0 变更)
2. **WebView2** - 灰屏 (未实现渲染)
3. **PDFium** - 滚动和渲染问题
4. **Windows Data PDF** - 完全不工作

---

## 问题分析与修复方案

### 1. MuPDF 引擎 (编译错误)

**错误1**: `Document::open(path)` - Path 类型不满足 `AsRef<FilePath>`
- **位置**: `src/engine/mupdf.rs:21`
- **原因**: mupdf 0.6.0 要求传入 `&str` 而非 `&Path`
- **修复**: 转换 Path 为 &str
```rust
// 旧
mupdf::Document::open(path)
// 新
mupdf::Document::open(path.to_str().unwrap())
```

**错误2**: `String` 上调用 `unwrap_or_else`
- **位置**: `src/engine/mupdf.rs:81`
- **原因**: `out.title` 是 `String` 不是 `Option`
- **修复**: 使用 `unwrap_or` 提供默认值
```rust
// 旧
let title = out.title.clone().unwrap_or_else(|| "Untitled".to_string());
// 新
let title = if out.title.is_empty() { "Untitled".to_string() } else { out.title.clone() };
```

**错误3**: `Outline` 没有 `page` 字段
- **位置**: `src/engine/mupdf.rs:83-86`
- **原因**: 0.6.0 API 变更，已移除 `page` 字段
- **修复**: 通过 `dest` 字段解析页码或设为 -1
```rust
// 旧
let page_index = out.page.clone().and_then(|loc| loc.page_number).unwrap_or(-1) as usize;
// 新 - 从 dest 解析页码或设为 0
let page_index = 0; // 需要通过 doc.resolve() 从 dest 解析
```

**错误4**: `Outline` 没有 `outlines()` 方法
- **位置**: `src/engine/mupdf.rs:88`
- **原因**: 0.6.0 API 变更，改用 `down` 字段
- **修复**: 使用 `out.down` 访问子项
```rust
// 旧
let children = if let Ok(sub_outlines) = out.outlines() { ... }
// 新
let children = self.parse_outline(out.down.clone(), level + 1);
```

**完整 parse_outline 修复**:
```rust
fn parse_outline(&self, outlines: Vec<mupdf::Outline>, level: usize) -> Vec<TocItem> {
    let mut items = vec![];
    for out in outlines {
        let title = if out.title.is_empty() { "Untitled".to_string() } else { out.title.clone() };
        let page_index = 0; // dest解析较复杂，暂设0
        
        // 使用 down 字段访问子项
        let children = if !out.down.is_empty() {
            self.parse_outline(out.down.clone(), level + 1)
        } else {
            vec![]
        };

        items.push(TocItem {
            title,
            page_index,
            children,
            level,
        });
    }
    items
}
```

### 2. WebView2 引擎 (灰屏)

**问题**: 引擎只存储路径，`render_page` 返回错误，无实际渲染实现

**分析**: WebView2 需要原生窗口覆盖，无法通过 `render_page` 接口返回 CPU 位图

**方案A (推荐)**: 实现真正的渲染 - 使用 webview2-com 渲染到内存位图
- 缺点: WebView2 本身不提供将渲染内容导出为位图的 API
- 替代: 可以用 Edge WebView2 的 PrintToPdf 功能

**方案B**: 移除或标记为不兼容
- 因为 WebView2 是浏览器覆盖层，无法融入 egui 的纹理系统

**采用的修复**:
1. 修改 `render_page` 返回一个占位图像或提示信息
2. 在 UI 中添加特殊处理：当选择 WebView2 时显示提示
3. 或者完全禁用此引擎选项

```rust
fn render_page(&self, _page_index: usize, _zoom: f32) -> Result<ColorImage> {
    // 返回一个提示图像
    let width = 600;
    let height = 800;
    let mut pixels = vec![128u8; width * height * 4]; // 灰色占位
    // 添加文字 "WebView2"
    Ok(ColorImage::from_rgba_unmultiplied([width, height], &pixels))
}
```

或者更彻底的方案：在 main.rs 中检测到 WebView2 时显示特殊 UI

### 3. PDFium 引擎 (滚动和渲染问题)

**问题1: 滚动计算**
- `main.rs:342` 固定使用 `row_height = 810.0 * zoom`
- PDF 页面实际高度可能不同

**问题2: 渲染尺寸**
- `pdfium.rs:47-49`: 使用 `zoom * 1.33` 进行 DPI 转换
- 1.33 是 96/72 = 1.33，这是标准转换，但可能与实际不符

**修复**:
1. 每个页面使用实际渲染高度计算滚动
2. 修正 zoom 计算逻辑: zoom 直接作为缩放因子而不是 DPI 乘数

```rust
// pdfium.rs - render_page 修复
fn render_page(&self, page_index: usize, zoom: f32) -> Result<ColorImage> {
    let doc = self.document.as_ref().ok_or_else(|| anyhow!("文档未打开"))?;
    let page = doc.pages().get(page_index as i32)?;

    // 使用页面原始尺寸，不做 DPI 转换
    let width_pt = page.width().value;
    let height_pt = page.height().value;

    // 直接使用 zoom 作为缩放
    let target_width = (width_pt * zoom) as i32;
    let target_height = (height_pt * zoom) as i32;

    let render_config = PdfRenderConfig::new()
        .set_target_width(target_width)
        .set_target_height(target_height);

    let bitmap = page.render_with_config(&render_config)?;
    // ... 其余代码
}
```

**滚动修复** (main.rs):
```rust

// 在 request_render 时返回页面高度信息
// 或者在 texture 加载后获取实际大小
```

### 4. Windows Data PDF 引擎 (完全不工作)

**问题分析**:
1. `win_pdf.rs:31`: `PdfDocument::LoadFromFileAsync(&file)?.get()?`
   - Windows async API 在同步上下文中可能失败
   
2. `win_pdf.rs:52`: `RenderToStreamAsync` 可能返回错误
3. `win_pdf.rs:57-59`: DataReader 读取可能失败
4. `win_pdf.rs:63`: `image::load_from_memory` 对 Windows 输出的图像格式可能不兼容

**修复**:

```rust
fn open(&mut self, path: &Path) -> Result<()> {
    let abs_path = std::fs::canonicalize(path)?;
    let path_str = abs_path.to_string_lossy();
    
    // 使用 blocking 方式处理 async
    let file = std::thread::spawn(move || {
        StorageFile::GetFileFromPathAsync(&windows::core::HSTRING::from(path_str.as_ref()))
    }).join().map_err(|e| anyhow!("线程错误: {:?}", e))??;
    
    let doc = std::thread::spawn(move || {
        PdfDocument::LoadFromFileAsync(&file)
    }).join().map_err(|e| anyhow!("加载错误: {:?}", e))??.get()?;
    
    self.document = Some(doc);
    Ok(())
}
```

但这有个问题：windows-rs 的 async 使用 Windows 的 IAsyncOperation，需要 COM 初始化

**更可靠的方案**: 使用 Windows.Foundation 的异步支持

或者更简单的方案：由于 Windows PDF API 比较复杂，可以暂时禁用或标记为实验性功能

**采用的修复**:
1. 添加更详细的错误处理
2. 简化渲染流程
3. 如果无法工作，记录日志并返回错误

```rust
fn render_page(&self, page_index: usize, zoom: f32) -> Result<ColorImage> {
    let doc = self.document.as_ref().ok_or_else(|| anyhow!("文档未打开"))?;
    let page = match doc.GetPage(page_index as u32) {
        Ok(p) => p,
        Err(e) => return Err(anyhow!("获取页面失败: {:?}", e)),
    };

    let stream = match InMemoryRandomAccessStream::new() {
        Ok(s) => s,
        Err(e) => return Err(anyhow!("创建流失败: {:?}", e)),
    };
    
    let options = match PdfPageRenderOptions::new() {
        Ok(o) => o,
        Err(e) => return Err(anyhow!("创建选项失败: {:?}", e)),
    };
    
    // 设置渲染尺寸 - 直接使用 zoom
    let size = page.Size().map_err(|e| anyhow!("获取尺寸失败: {:?}", e))?;
    options.SetDestinationWidth((size.Width * zoom) as u32).map_err(|e| anyhow!("设置宽度失败: {:?}", e))?;
    options.SetDestinationHeight((size.Height * zoom) as u32).map_err(|e| anyhow!("设置高度失败: {:?}", e))?;

    // 渲染 - 使用 block_on 或类似机制
    // 由于 windows-rs 的 async 设计，这步可能需要 runtime
    page.RenderToStreamAsync(&stream).map_err(|e| anyhow!("渲染失败: {:?}", e))?
        .get().map_err(|e| anyhow!("等待渲染失败: {:?}", e))?;

    // 读取数据
    let size_u64 = stream.Size().map_err(|e| anyhow!("获取流大小失败: {:?}", e))?;
    let mut buffer = vec![0u8; size_u64 as usize];
    
    let reader = windows::Storage::Streams::DataReader::CreateDataReader(&stream)
        .map_err(|e| anyhow!("创建读取器失败: {:?}", e))?;
    reader.LoadAsync(size_u64 as u32).map_err(|e| anyhow!("加载数据失败: {:?}", e))?
        .get().map_err(|e| anyhow!("等待数据失败: {:?}", e))?;
    reader.ReadBytes(&mut buffer).map_err(|e| anyhow!("读取失败: {:?}", e))?;

    // Windows PDF 可能输出 BMP，需要检测格式
    let img = image::load_from_memory(&buffer)
        .or_else(|_| {
            // 可能是 BMP，手动解析
            Err(anyhow!("无法解码图像"))
        })?;
    
    let rgba = img.to_rgba8();
    let (w, h) = rgba.dimensions();
    Ok(ColorImage::from_rgba_unmultiplied([w as usize, h as usize], rgba.as_raw()))
}
```

如果 async 处理太复杂，可以考虑完全禁用此引擎，因为其他引擎已经可用。

## 实施顺序

1. **首先修复 MuPDF** - 这是用户首选引擎之一，必须能编译
2. **其次修复 PDFium** - 这是默认引擎，修复滚动/渲染问题
3. **然后处理 Windows Data PDF** - 如果太复杂可以暂时禁用
4. **最后处理 WebView2** - 选择实现或禁用
